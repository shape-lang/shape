## §1. Root-cause analysis

`lower_list_comprehension_expr`
(`crates/shape-vm/src/mir/lowering/expr.rs:364`) pre-W11-fup-B path:

1. Evaluate `clause.iterable` start/end into discarded temps (W11-jit-
   new-array's narrow Range carrier deletion, `836a77f5`).
2. Allocate `element_slot` via `alloc_temp(LocalTypeInfo::Unknown)`.
3. `assign_none(element_slot, ...)` — write `Use(Constant(None))` to
   the slot.
4. `lower_destructure_bindings_from_place(pattern, &Place::Local(
   element_slot))` — allocates the user-visible iteration variable
   `x` via `alloc_local(name, LocalTypeInfo::Unknown)` and emits
   `Assign(x_slot, Use(Copy(element_slot)))`.

The JIT kind-tracker (`crates/shape-jit/src/mir_compiler/types.rs:289
::infer_slot_kinds_with_concrete`) forward-pass inheritance chain:

- `element_slot` ← `Use(Constant(None))` → `infer_constant_kind(None)`
  returns `None` (`types.rs:1538`, no fabricated default per ADR-006
  §2.7.7 #9).
- `x_slot` ← `Use(Copy(element_slot))` → inherits `element_slot`'s
  kind = `None`.

Downstream `Assign(temp, BinaryOp(Mul, Copy(x_slot), Copy(x_slot)))`
in the comprehension body reaches `compile_rvalue`'s BinaryOp arm with
both operand kinds = `None` → falls through the typed fast paths
(`compile_binop_f64`, `compile_binop_int64`, `compile_binop_bool`) to
`compile_binop` (generic), then to `compile_binop_dynamic_arith` →
surface-and-stop. The kind-erasure site is the `assign_none` write
combined with the destructure-pattern-based binding, which is
structurally incapable of threading the iteration variable's kind
when the iterable is a closed range yielding scalars.

Mirror comparison: `lower_for_expr`
(`crates/shape-vm/src/mir/lowering/expr.rs:952`) Range case allocates
the user-visible loop variable AS the counter slot and seeds it via
`Use(Copy(start_slot))` where `start_slot` is the result of
`lower_expr_to_temp(start_expr)` — for the common `0..N` form the
start expression's `Int` literal stamps `start_slot`'s kind as
`Int64`, which inherits through the `Use(Copy(_))` to the counter
slot. The bytecode-side counterpart (`crates/shape-vm/src/compiler/
loops.rs:34::try_begin_range_counter_loop`) does the same shape:
`declare_local(var_name)` for the counter, `compile_expr(start_expr)`
then `StoreLocal(counter_local)` — bytecode + MIR agree on the
counter-binding shape.

The MIR comprehension lowering missed this mirror; W11-fup-B adds it.

## §2. Files changed

### `crates/shape-vm/src/mir/lowering/expr.rs::lower_list_comprehension_expr`

Lines 364-484 (post-fix). Added an `iv_counter_simple_range` selector
that matches when the clause pattern is a simple identifier and the
iterable is `Expr::Range { start: Some(_), end: Some(_), .. }`. In
that case:

- `let start_slot = lower_expr_to_temp(builder, start_expr);` — lower
  the start expression into a temp slot (the kind-source).
- `let _ = lower_expr_to_temp(builder, end_expr);` — lower the end
  expression for the borrow tracker; the value itself is not used by
  the comprehension's downstream MIR shape (mirrors W11-jit-new-array
  Range-carrier deletion).
- `let iv_slot = builder.alloc_local(iv_name, LocalTypeInfo::Copy);`
  — allocate the user-visible iteration variable as a named local
  (mirroring `lower_for_expr`'s counter binding).
- `assign_copy_from_slot(builder, iv_slot, start_slot, ...);` — seed
  the iteration variable from the start_slot. The JIT forward kind-
  inference pass propagates `Int64` (from the int literal) through
  this `Use(Copy(start_slot))` write.

The non-identifier-pattern / non-Range fallback path is preserved
verbatim (W11-jit-new-array Range-carrier deletion intact for
borrow-tracker compatibility; the broader iterable-element-kind
threading is out of scope per the narrow W11-fup-B charter — see §4).

Per ADR-006 §2.7.5 stamp-at-compile-time: the iteration variable's
kind source IS the start expression's literal; no runtime tag dispatch
is reintroduced; no fabricated default; the existing forward kind-
inference pass handles propagation without modification to the JIT.

No other files changed.

## §4. Out-of-scope / surface-and-stop residuals

### Iteration-var kind for non-Range iterables (broader scope)

The W11-fup-B fix is bounded to `Expr::Range { start: Some, end:
Some }` iterables — array-literal / function-call / variable-bound /
generic-Iterable iterables fall through to the original `element_slot
← Constant(None)` path. The dispatch explicitly notes:

> If the comprehension iteration-var kind comes from a generic
> Iterable trait whose return-element-kind is not threaded through
> the call signature, surface-and-stop — that's deeper architecture
> work outside W11-fup-B's narrow scope.

Surface site: `crates/shape-vm/src/mir/lowering/expr.rs:464-480`
(fallback path's `assign_none` write to `element_slot`). Broader
architectural fix would require either:

(a) Plumbing the iterable's `ConcreteType` element (from the
bytecode compiler's type tracker / `concrete_seed`) into the MIR
lowering at the element_slot allocation site, OR

(b) Extending the JIT kind-tracker to project through the iteration
variable's element type at the comprehension MIR shape directly
(mirrors §2.7.5 producer-side classification but at the consumer
side).

Tracked as `W11-followup-comprehension-iterable-element-kind` (NEW —
distinct from W11-fup-B's Range-only narrow scope).
