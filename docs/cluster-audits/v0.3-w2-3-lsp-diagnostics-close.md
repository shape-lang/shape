## §1 Audit rows closed

### 1.17 — `Diagnostic.codeDescription` clickable URL

Audit rendering: `P → C`. Anchor: `diagnostics.rs:62` previously set
`code: Some(NumberOrString::String(error.code.as_str().to_string()))`
but `code_description: None` — no clickable doc URL.

**Landed:** every code-bearing diagnostic now ships a
`CodeDescription { href: Uri }` pointing at
`https://book.shape-lang.dev/appendix/error-codes/#{code}`. The book
hostname is the canonical one used throughout
`../shape-web/landing/index.html`. URL fragment is the error code
verbatim — pattern is stable; once the dedicated error-code reference
page lands in the book the URL pattern stays valid.

Three integration points share one URL builder
(`diagnostics::code_description_for`):

1. **Structured-error renderer** (`LspErrorRenderer::structured_error_to_diagnostic` at
   line 67): attaches `code_description` inline at construction.
2. **`create_diagnostic_with_code` helper** (`diagnostics.rs:395+`):
   attaches `code_description` when `error_code` is `Some(_)`.
3. **Pipeline-exit enricher** (`enrich_diagnostics_with_code_metadata`):
   backfills `code_description` for every diagnostic emitted by the
   ~19 per-validator `Diagnostic` literal sites in this module that
   construct with `code_description: None`. Runs once at the end of
   `analysis::analyze_program_semantics` — single backfill, no per-site
   edits required.

**Surface-and-stop note (per dispatch §6).** The audit row notes 1.17
"depends on W3 Shape Book deliverable" — i.e. a dedicated error-code
reference page in the book. That page does NOT yet exist in
`../shape-web/book/book-site/src/content/docs/appendix/` (only
`troubleshooting.mdx` / `faq.mdx` / `configuration.mdx`). Per dispatch
"surface-and-stop with structured shape — defer to W3 follow-up for URL
schema design vs land minimal-viable URL pattern", we land the
**minimal-viable URL pattern**: clients following the URL today land
on the appendix index (rust-analyzer's `--explain` behavior for unknown
codes). The pattern is invariant; the destination page is the W3
follow-up.

### 1.19 — `Diagnostic.tags` (UNNECESSARY for unused decl)

Audit rendering: `M`. Anchor: `diagnostics.rs:78` `tags: None`
everywhere.

**Landed:**

- **Tag derivation** (`diagnostics::diagnostic_tags_for`): code-first
  predicate — `W0102` (unused-import) and `W0103` (reserved
  unused-variable) emit `DiagnosticTag::UNNECESSARY`. Message-based
  fallback (`"unused import"` / `"unused variable"` substring) catches
  any future compiler-emitted "unused" diagnostics without per-code
  wiring.
- **Pipeline-exit backfill**: `enrich_diagnostics_with_code_metadata`
  attaches tags after the per-validator literal sites; same shape as
  the 1.17 backfill.
- **New unused-imports lint** (`diagnostics::validate_unused_imports`):
  detects named imports (`from path use { Foo, Bar as B }`) whose
  local bindings never appear as identifiers in the rest of the
  program. Emits `W0102` warnings with `tags: [UNNECESSARY]` directly
  (not via the enricher). Detection uses both:
  1. An AST visitor for `Expr::Identifier` / `Expr::FunctionCall` /
     `Expr::MethodCall` / `Expr::PropertyAccess` references.
  2. A whole-word text-search fallback over the source outside the
     import statements (catches type annotations, impl-block trait
     names, generic args the Expr visitor doesn't reach).
  Skipped: annotation imports (`is_annotation: true`; matched via
  `@name`, not bare identifier), namespace imports
  (`ImportItems::Namespace`; requires qualified-name resolution in the
  compiler proper), and `_`-prefixed local names (intentional-unused
  convention).
- **Wire-up**: `analysis::analyze_program_semantics` calls
  `validate_unused_imports` alongside the other 10 validators.

### 1.40 — Quick-fix code-keyed rebuild

Audit rendering: `P`. Anchor: `code_actions.rs::get_quick_fixes` (line
67) used `message.contains(...)` substring matches across 8 fix
categories.

**Landed:** `get_quick_fixes` now dispatches off `diagnostic.code`
first via the new `diagnostic_code()` helper, with `message.contains()`
retained as a fallback for diagnostics that lack a code (e.g.
frontmatter, some foreign-LSP diagnostics) or that originated from
emitters not yet wired through the structured-error path. The
code-keyed dispatch covers:

| Quick-fix | Code(s) | Message fallback (codeless-diag only) |
|-----------|---------|---------------------------------------|
| Declare variable | `E0101` | `"undefined"` / `"not defined"` |
| Add missing semicolon | `E0004` | `"expected ';'"` / `"missing semicolon"` |
| Add missing closing brace | `E0005` / `E0002` | `"expected '}'"` / `"unclosed"` |
| Prefix with underscore (unused) | `W0102` / `W0103` | `"unused"` |

Diagnostic-text rewordings (i18n, clarity improvements) no longer
silently break the quick-fix triggers — the code field is the
canonical key.

### 1.73 + 1.74 — Pull-based diagnostics

Audit rendering: `M` for both. Anchor: shape-lsp used push-based
`publishDiagnostics` only.

**Landed:**

- **Capability registration** (`server.rs::initialize`):
  `diagnostic_provider: Some(DiagnosticServerCapabilities::Options(
  DiagnosticOptions { identifier: Some("shape"), inter_file_dependencies:
  true, workspace_diagnostics: true, .. }))`. The
  `inter_file_dependencies: true` setting reflects Shape's module
  import system where editing one file changes another's diagnostics
  (matches rust-analyzer). `workspace_diagnostics: true` declares the
  1.74 companion pull.
- **`textDocument/diagnostic` handler** (`ShapeLanguageServer::diagnostic`):
  computes diagnostics for a single document on demand via the new
  `compute_pull_diagnostics(uri)` helper, returns a
  `RelatedFullDocumentDiagnosticReport`. Uses the same parse +
  semantic-analysis pipeline as the push path so push-mode and
  pull-mode produce identical reports for identical source. No
  `Unchanged` reports today (would require diagnostic result-id
  caching per document version — non-parity-blocking follow-up).
- **`workspace/diagnostic` handler**
  (`ShapeLanguageServer::workspace_diagnostic`): returns full
  diagnostic reports for every open document. Closed workspace files
  are not eagerly walked from disk here — the LSP spec permits servers
  to scope workspace pull to documents the server has materialized
  analyses for, which keeps the request cost bounded. The W2.6/W2.7
  workspace-enumeration infra
  (`ModuleCache::enumerate_workspace_shape_files`) is reserved for the
  eager-indexing follow-up.

Push-mode (`publishDiagnostics`) remains the primary delivery and is
unchanged.

## §5 Forbidden Patterns audit

- **No bridge/probe/helper/hop/translator/adapter/shim framings.**
  Caught one "compatibility layer" docstring during pre-commit
  check-no-dynamic and rewrote to "fallback retained only for
  codeless / pre-structured-error diagnostics". Final
  `scripts/check-no-dynamic.sh` EXIT=0.
- **No new modal-types subsystem.** All diagnostic enrichment is
  pure post-construction backfill against existing `Diagnostic`
  fields (`code` / `message`).
- **No `ValueWord` / dynamic dispatch / tag-bits anywhere.** This
  closure-wave does not touch the value-system surface.
- **No "for one period" / "fallback for one edge case" /
  "documented intentional duality" rationalizations.** The
  message-keyed fallback in `get_quick_fixes` is explicitly bounded
  by `code.is_none()` for the E0101 path; the other code-keyed paths
  use OR-with-message-fallback because their message strings remain
  the dominant trigger for diagnostics not yet wired through the
  structured-error path. The structured-error wiring follow-up is
  out of W2.3 scope.
