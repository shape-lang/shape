## §1 Systemic divergences (cross-cutting; affect most pages)

These four patterns affect the majority of pages in slice B. Listed once at top, then per-page divergences below assume them and note only page-specific issues.

### §1.A `::` namespacing vs `.` dot dispatch (every native + core page)

**Book pattern:** `crypto::sha256(...)`, `io::open(...)`, `file::read_text(...)`, `set::new()`, `json::parse(...)`, etc.

**HEAD pattern:** All stdlib-src declarations + tests use dot dispatch — `crypto.sha256(...)`, `io.open(...)`, `file.read_text(...)`, `set.new()`, `json.parse(...)`.

**Cite:**
- `crates/shape-runtime/stdlib-src/core/crypto.shape:12-14` `let hash = crypto.sha256("hello")`
- `bin/shape-cli/tests/stdlib/stdlib_new_modules.rs:264` `use std::core::crypto` then `crypto.sha256(...)` form
- Grammar: `crates/shape-ast/src/shape.pest:64` `from module_path use { ... }` is supported; module path uses `::`, but member access on a bound module name uses `.`.

**Disposition:** book-wrong (W3.2-B fix-wave doc-only; mechanical s/<mod>::<fn>/<mod>.<fn>/g over `stdlib/` pages).

### §1.B Canonical import path `std::core::<mod>` vs book `std::<mod>` (every native page)

**Book pattern:** `use std::crypto`, `use std::http`, `use std::regex`, `use std::toml`, `use std::yaml`, `use std::xml`, `use std::archive`, `use std::compress`, `use std::env`, `use std::unicode`, `use std::parallel`.

**HEAD reality:** Every stdlib module is registered as `std::core::<name>`. No `std::<name>` alias exists.

**Cite:**
- `crates/shape-runtime/src/stdlib/crypto.rs:18` `ModuleExports::new("std::core::crypto")`
- `crates/shape-runtime/src/stdlib/crypto.rs:284` `assert_eq!(module.name, "std::core::crypto")`
- All tests use `use std::core::crypto` form: `bin/shape-cli/tests/stdlib/stdlib_new_modules.rs:264,276,288,300,312`
- grep finds zero occurrences of `std::crypto` (without `::core::`) in tests/examples/stdlib-src.
- Book file/io/core pages already use the correct full path (`use std::core::io`, `use std::core::file` at `stdlib/native/file.mdx:19`, `stdlib/native/io.mdx:13`, `stdlib/core/state.mdx:20`); only the 11 short-form native pages diverge.

**Disposition:** book-wrong (W3.2-B fix-wave doc-only; mechanical s/use std::/use std::core::/g over the 11 short-form native pages).

### §1.C `Result<T>` (single-arg) vs `Result<T, E>` (two-arg) (most native pages)

**Book pattern:** `Result<string>`, `Result<HashMap>`, `Result<Vec<string>>`, `Result<()>`, `Result<Json>`, `Result<T>`.

**HEAD reality:** `Result<T, E>` two-arg only. `crates/shape-runtime/stdlib-src/core/result_methods.shape:7` declares `extend Result<T, E>`. All stdlib `builtin fn` declarations return `Result<_, string>` form (see compress/file/http/json/toml/xml/yaml/regex/crypto).

**Cite (book):**
- `stdlib/native/crypto.mdx:65` `crypto.base64_decode(encoded: string) -> Result<string>`
- `stdlib/native/file.mdx:82-87` `(string) -> Result<string>`, `Result<()>`, `Result<Vec<string>>`, `Result<Vec<number>>`
- `stdlib/native/http.mdx:23` `Result<HttpResponse>`
- `stdlib/native/json.mdx:256-260` `Result<Json>`, `Result<Type>`, `Result<string>`
- `stdlib/native/toml.mdx:22,38` `Result<HashMap>`, `Result<string>`
- `stdlib/native/yaml.mdx:22,36,53` `Result<HashMap>`, `Result<Array>`, `Result<string>`
- `stdlib/native/xml.mdx:33,64` `Result<HashMap>`, `Result<string>`

**Cite (HEAD):**
- `crates/shape-runtime/stdlib-src/core/crypto.shape:134,168` `Result<string, string>`
- `crates/shape-runtime/stdlib-src/core/file.shape:36,54,71,89,106,124` all `Result<..., string>`
- `crates/shape-runtime/stdlib-src/core/http.shape:39,60,81,102,123,146,168,186` all `Result<_, string>`
- `crates/shape-runtime/stdlib-src/core/json.shape:44,78` `Result<_, string>` / `Result<string, string>`

**Disposition:** book-wrong (W3.2-B fix-wave doc-only); OR (if "single-arg Result with implicit string E" is intended as language sugar) the v0.3 fix-wave needs to add this sugar and surface it in grammar — currently it does NOT parse. Default disposition: book-wrong.

### §1.D `pub struct` (book) vs `pub type` (Shape syntax)

**Book pattern:** `pub struct FunctionRef { ... }`, `pub struct Frame { ... }`, `pub struct VmState { ... }`, `pub struct CallPayload { ... }`, `pub struct Delta { ... }` — all in `stdlib/core/state.mdx:34,49,64,84,97,109,122`.

**HEAD reality:** Shape has NO `struct` keyword. Grammar uses `type Name { ... }` for struct-shaped types. `stdlib-src/core/state.shape:10,20,31,38,46,53,61` declares them all as `pub type FunctionRef { ... }`, `pub type Frame<T> { ... }`, etc.

**Cite (grammar):** `crates/shape-ast/src/shape.pest:120-124` — rule is named `struct_type_def` internally but the keyword is `type`, not `struct`.

**Disposition:** book-wrong (W3.2-B fix-wave doc-only; mechanical s/pub struct/pub type/g over state.mdx).

## §6 Candidate imprecisions (forwarded to imprecision-log)

1. **Imprecision candidate B1:** `stdlib/core/intrinsics` import path used by `core/collections.mdx:19` (`from std::core::intrinsics use { HashMap }`) — verify whether this is the canonical user-facing path for `HashMap` or whether the intrinsics namespace is supposed to be compiler-internal (CLAUDE.md "Intrinsics gated" notes `__intrinsic_*` gating, but says `HashMap` is among the "Builtins & Intrinsics" section).
2. **Imprecision candidate B2:** `Result<T>` single-arg form — book uses this on 7+ pages. Either (a) grammar/inference needs to accept this as sugar for `Result<T, string>` (the dominant stdlib E type) — would unblock book accuracy + match user expectations from Rust — or (b) book is uniformly wrong. Decision binary; needs explicit team-lead ruling.
3. **Imprecision candidate B3:** `pub struct` vs `pub type` — book uses `struct` keyword in state.mdx. Either grammar should accept `struct` as alias for `type` (the underlying AST rule IS named `struct_type_def`) or book is wrong. Currently book is wrong but the user-expectation (struct=type with named fields) is universal — worth considering grammar alias.
4. **Imprecision candidate B4:** `Any` keyword — book uses `Array<Any>`, `Map<string, Any>` in state.mdx. Shape grammar has NO `Any` keyword (CLAUDE.md: "No `any` type"). `stdlib-src/core/json_value.shape:22` uses lowercase `any` in `Json::Array(any)` — verify what `any` resolves to in type system. Book using capital `Any` consistent with Rust/TS expectations.
5. **Imprecision candidate B5:** `simulation` module classification — book overview lists it under "Domain Libraries" but actual stdlib module path is `std::core::simulation` (core, not domain). Same for `simulation.mdx` file itself living under `domain/`. Either move book file to `core/` OR rename stdlib module to `std::domain::simulation`. (Recommended: move book file; stdlib path is the source of truth and `simulation.shape` correctly identifies itself as `@module std::core::simulation` at `stdlib-src/core/simulation.shape:1`.)
6. **Imprecision candidate B6:** Several stdlib pages declare module-level functions with REQUIRED final arguments that the book documents as OPTIONAL: `compress.zstd(data, level)` (book: `level?`), `json.stringify(value, pretty)` (book: optional). Either stdlib should be updated to use `= default` syntax (`stdlib-src/core/utils/testing.shape:16` shows `condition, message = "..."` works), OR book is wrong. Stdlib-update is the cleaner fix.
