## §1. Root cause analysis

The MIR-time kind label for `ConcreteType::Array(_)` is `NativeKind::Ptr(HeapKind::TypedArray)` per `crates/shape-jit/src/mir_compiler/types.rs:175`:

```rust
ConcreteType::Array(_) => NativeKind::Ptr(HeapKind::TypedArray),
```

The runtime carrier is the v2-raw `*mut TypedArray<T>` pointer produced by `crate::ffi::v2::jit_v2_array_new_<kind>` allocators — NOT `Arc<TypedArrayData>` (that enum + outer `HeapValue::TypedArray(Arc<_>)` arm + `TypedBuffer<T>` wrapper layer were retired across V3-S5 ckpt-1..ckpt-4 per W12-typed-array-data-deletion audit §3.5 + §3.6 + ADR-006 §2.7.24 Q25.A SUPERSEDED). `HeapKind::TypedArray = 8` is the vacated-ordinal type LABEL.

The print Call-terminator dispatch at `crates/shape-jit/src/mir_compiler/terminators.rs` had no arm for `Some(NativeKind::Ptr(HeapKind::TypedArray))`. The `_`-arm SURFACEd with the structured message above.

**Bonus pre-existing gap**: even after adding the FFI body that delegates to the canonical `ValueFormatter::format_kinded` via `NativeKind::UInt64` (the carrier-recognition label `as_v2_typed_array` accepts at `crates/shape-vm/src/executor/v2_handlers/v2_array_detect.rs:181-216`), the JIT-side allocators (`jit_v2_array_new_<f64,i64,i32,bool,string,decimal>`) were NEVER stamping the element-type byte at HeapHeader `_pad` offset 7. The VM-side `op_new_typed_array_*` handlers stamp it (e.g. `crates/shape-vm/src/executor/v2_handlers/array.rs:55-58`):

```rust
let ptr = TypedArray::<i64>::with_capacity(cap);
unsafe { stamp_elem_type(ptr as *mut u8, ELEM_TYPE_I64) };
self.push_kinded(ptr as usize as u64, NativeKind::UInt64)?;
```

The JIT-side allocators did `TypedArray::<i64>::with_capacity(capacity)` and returned the raw pointer — leaving `_pad = 0 = ELEM_TYPE_UNKNOWN`. So `as_v2_typed_array(bits, NativeKind::UInt64)` returned `None`, the formatter fell through to `slot.as_u64().to_string()`, and the print output was the raw pointer printed as a u64 scalar (`100218864737360`).

The fix has two parts to close this end-to-end.

## §2. Files changed

### `crates/shape-jit/src/ffi/conversion.rs::jit_print_typed_array`

New FFI body `(ctx_ptr: *const JITContext, bits: u64) -> ()`:

```rust
#[unsafe(no_mangle)]
pub extern "C" fn jit_print_typed_array(
    ctx_ptr: *const crate::context::JITContext,
    bits: u64,
) {
    if bits == 0 {
        println!("None");
        return;
    }
    let registry = registry_from_ctx(ctx_ptr);
    let formatter = shape_vm::executor::printing::ValueFormatter::new(&registry);
    // Reify with NativeKind::UInt64 — the carrier-recognition label the
    // canonical formatter uses to route v2-raw *mut TypedArray<T>
    // pointers to its format_v2_typed_array per-element walker at
    // printing.rs:133-137.
    let slot = shape_value::ValueSlot::from_raw(bits);
    let kinded = shape_value::KindedSlot::new(slot, shape_value::NativeKind::UInt64);
    let rendered = formatter.format_kinded(&kinded);
    std::mem::forget(kinded);
    println!("{}", rendered);
}
```

Per ADR-006 §2.7.5 stamp-at-compile-time: the kind label IS the discriminator. The `Ptr(HeapKind::TypedArray)` MIR-time label routes to this FFI; the FFI re-labels with `UInt64` for the canonical formatter's v2-raw carrier-recognition path. No `is_heap()` probe; no kind from raw bits; no Bool-default fabrication.

### `crates/shape-jit/src/ffi_refs.rs`, `ffi_builder.rs`, `ffi_symbols/object_symbols.rs`

Standard FFI plumbing — `print_typed_array: FuncRef` field + `r!()` initializer + `builder.symbol("jit_print_typed_array", ...)` + Cranelift signature `(I64, I64) -> ()`.

### `crates/shape-jit/src/mir_compiler/terminators.rs`

New arm in the print Call-terminator dispatch (placed after `Some(NativeKind::Ptr(HeapKind::Iterator))` per cw-D-fam3 Collection-family ordering):

```rust
Some(NativeKind::Ptr(HeapKind::TypedArray)) => {
    let val_ty = self.builder.func.dfg.value_type(val);
    let widened = if val_ty == types::I64 {
        val
    } else if val_ty == types::F64 {
        self.builder.ins().bitcast(types::I64, MemFlags::new(), val)
    } else {
        val
    };
    self.builder.ins().call(
        self.ffi.print_typed_array,
        &[self.ctx_ptr, widened],
    );
}
```

### `crates/shape-jit/src/ffi/v2/mod.rs` (6 allocators)

Stamped each `jit_v2_array_new_<f64,i64,i32,bool,string,decimal>` allocator's HeapHeader `_pad` byte immediately after `TypedArray::<T>::with_capacity(...)`. Example:

```rust
#[unsafe(no_mangle)]
pub extern "C" fn jit_v2_array_new_i64(capacity: u32) -> *mut TypedArray<i64> {
    let ptr = TypedArray::<i64>::with_capacity(capacity);
    unsafe { stamp_elem_type(ptr as *mut u8, ELEM_TYPE_I64) };
    ptr
}
```

Mirror of VM-side `op_new_typed_array_*` handler pattern at `crates/shape-vm/src/executor/v2_handlers/array.rs:40-81`. Per ADR-006 §2.7.5 stamp-at-compile-time: the element kind is statically proven at the JIT codegen site (`mir_compiler/v2_array.rs:166-176` dispatch table picks the matching `jit_v2_array_new_<kind>` FuncRef per the producing slot's `NativeKind`), so the stamp byte authoritatively records that compile-time-proven element kind into the v2-raw allocation's heap-header — same compile-time-proof shape the VM-side handlers use.

### `crates/shape-vm/src/executor/mod.rs` + `crates/shape-vm/src/executor/v2_handlers/mod.rs`

Exposed `pub mod v2_handlers;` + `pub mod v2_array_detect;` so the JIT-side allocator stamps can call `stamp_elem_type` + the `ELEM_TYPE_*` constants. The `printing` module was already `pub mod` (the W12-jit-print-heap-arm-classification family of FFI bodies already use it). SoC: the JIT-side allocators are the only callers gaining new pub access.
