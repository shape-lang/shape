## §6 Candidate audit-text imprecisions caught during pre-flight ground-truth

Per Reading 6 binding ("every claim grep-verified at HEAD"), surfacing imprecisions in audit text I produced — and in the parent dispatch / related ADRs / CLAUDE.md text — that this audit's grep cycle caught:

1. **`int` representation contradiction (cross-doc)** — this audit's §1 page #10 + page #11 reconciliation. `integer-types.mdx` says `int` is "signed 48-bit", `builtin-types.mdx` says "Signed 64-bit", `CLAUDE.md` line 75 says "`int` (i48)". The single ground truth is `concrete_conv.rs:277`: `"int" => ConcreteType::I64`. The "i48" framing is a **historical NaN-boxed payload size** that the strict-typing migration superseded (per CLAUDE.md "Forbidden Patterns" — `ValueWord` deleted). **Imprecision flagged in BOTH the book AND CLAUDE.md.** CLAUDE.md line 75 should be updated to reflect that `int` is i64 (typed) — but per dispatch binding "CLAUDE.md modifications require explicit user authorization", this is **FLAGGED EXPLICITLY** rather than self-edited.

The `llm_common_mistakes` framing is double-wrong: it implies tuples-as-`(a, b)` exist but can't be tail-expr (they don't exist syntactically) AND it implies tuple-destructuring doesn't work (correct — it doesn't because the syntax doesn't exist). Should be: "Shape's tuple types use bracket syntax `[T1, T2, T3]`. There is no `(a, b)` tuple form." — landed in W3.2-A.

3. **CLAUDE.md "Pipe operator" claim (line 91)** — CLAUDE.md says "Pipe operator: `expr |> fn`". Grammar confirms `BinaryOp::Pipe` exists at `ast/operators.rs:53`. Book `operators.mdx` does NOT mention `|>` at all — divergence noted in §1 page #2 + §2.A. CLAUDE.md is accurate; book is incomplete.

5. **Stale "NaN-boxed" framing leaking into user docs** — `integer-types.mdx` lines 10, 30-37, 105, 107 all reference NaN-boxed representation.
