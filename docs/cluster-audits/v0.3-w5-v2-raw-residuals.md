### §1.D Producer/consumer chain — `clone_slot_kinded` share-accounting

Pre-fix shape of `clone_slot_kinded` (at
`crates/shape-vm/src/executor/vm_state_snapshot.rs:295-300`):

```rust
fn clone_slot_kinded(bits: u64, kind: NativeKind) -> KindedSlot {
    let carrier = KindedSlot::new(ValueSlot::from_raw(bits), kind);
    carrier.clone()
}
```

`KindedSlot::new` (kinded_slot.rs:80) is a pure wrapper — it does NOT bump
the underlying refcount. The constructor's contract requires the caller to
have already transferred a share into the carrier. Here, no share was
transferred — the bits came from a live VM-owned slot whose share remains
owned by the VM.

Step trace for a `Ptr(HeapKind::TypedObject)` slot at refcount=1 owned by
the live VM module-binding:

1. `KindedSlot::new(bits, Ptr(TypedObject))` — carrier wraps the bits.
   Refcount unchanged at 1. **Both** the carrier and the live VM binding
   now claim ownership of the same share — double-claim bookkeeping bug.
2. `carrier.clone()` — `KindedSlot::Clone` for `Ptr(HeapKind::TypedObject)`
   calls `v2_retain` (kinded_slot.rs:1080-1084). Refcount 1→2. Returns
   the cloned KindedSlot.
3. Function returns the clone.
4. `carrier` drops at end-of-scope. `KindedSlot::Drop` for
   `Ptr(HeapKind::TypedObject)` calls `release_elem` →
   `v2_release` (kinded_slot.rs:[Drop arm]). Refcount 2→1.

Net effect on refcount: 0 (Clone+Drop cancel). But the returned `cloned`
KindedSlot now owns 1 conceptual share — a share that the underlying
refcount never accounted for.

When the snapshot drops later (at end of `invoke_module_fn_id_stub`), the
cloned's `KindedSlot::Drop` retires its share → v2_release → refcount
1→0 → `_drop` runs and deallocates. The live VM binding still references
the freed memory. Subsequent reads (debug print, VM drop walking the
bindings, another `clone_slot_kinded` for the same binding) are
use-after-free / double-free.

### §1.E Producer/consumer chain — `length_typed_object_empty` test fixture

Test fixture (pre-fix at
`crates/shape-vm/src/executor/objects/property_access.rs:619-635`):

```rust
let storage = TypedObjectStorage::new(...);
let arc = Arc::new(storage);
let bits = Arc::into_raw(arc) as u64;
vm.push_kinded(bits, NativeKind::Ptr(HeapKind::TypedObject)).unwrap();
vm.op_length().unwrap();
let (len_bits, len_kind) = vm.pop_kinded().unwrap();
```

`op_length` retires the popped TypedObject share via `drop_with_kind`
(property_access.rs:554). For `Ptr(HeapKind::TypedObject)`,
`drop_with_kind` dispatches `TypedObjectStorage::release_elem`
(stack.rs:492-495). `release_elem` reads the embedded `HeapHeader`
refcount (initialized to 1 by `TypedObjectStorage::new`), decrements via
`v2_release` → 0 → runs `_drop`. `_drop` calls `std::alloc::dealloc(ptr,
Layout::new::<Self>())` — but `ptr` was produced by `Arc::into_raw`, NOT
by the matching v2-raw `_new` allocator. Wrong-allocator-pair free →
`free(): invalid size` SIGABRT.

Production code (e.g. `populate_module_objects` in
`vm_impl/modules.rs:433`, `build_entry_object` in
`objects/hashmap_methods.rs:179`) uses the v2-raw `_new` allocator
universally — so this allocator-pair mismatch never surfaces in shipped
code. The bug is **confined to test fixtures** that predate the
Wave-2-Agent-D1 v2-raw migration.

## §2 Bounded fix (Phase 2)

### §2.A Root cause classes + fix shape per class

**Class 1: share-accounting double-claim at the snapshot-clone boundary**
(w17_comptime_* SIGABRTs).

The fix mirrors the pattern at `executor/mod.rs:792-796::
module_binding_read_owned_kinded` — explicit `clone_with_kind` retain
followed by `KindedSlot::new` claim of the freshly minted share:

```rust
fn clone_slot_kinded(bits: u64, kind: NativeKind) -> KindedSlot {
    crate::executor::vm_impl::stack::clone_with_kind(bits, kind);
    KindedSlot::new(ValueSlot::from_raw(bits), kind)
}
```

`clone_with_kind` bumps the underlying refcount by 1 → that new share is
claimed by the returned `KindedSlot::new` wrapper. The live VM
binding/stack keeps its own share unchanged. Net effect: +1 share for the
snapshot's owner, the same as the Round 13 T5 / cluster-1.5 pattern just
applied to a snapshot-side boundary instead of a closure-call boundary.

**Class 2: test-fixture allocator-pair mismatch** (`length_typed_object_empty`
+ 5 sibling tests).

The fix migrates each affected test fixture from the legacy
`Arc::new(TypedObjectStorage::new(...))` + `Arc::into_raw(arc)` pattern
to the v2-raw `TypedObjectStorage::_new(...)` allocator. The v2-raw
allocator pairs with the v2-raw `_drop` deallocator used by the
production `drop_with_kind(Ptr(HeapKind::TypedObject))` dispatch arm.
Production code already uses `_new` everywhere; only stale tests still
used `Arc::new`.

Witness probes that previously asserted `Arc::strong_count` semantics
were rewritten to read the v2-raw refcount directly via
`(*ptr).header.refcount.load(Ordering::SeqCst)` — the canonical way to
inspect a `_new`-allocated TypedObject's share count.
