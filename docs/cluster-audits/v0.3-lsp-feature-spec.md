## §0 Scope, methodology, audit binding

### Scope

- Anchor source: `tools/shape-lsp/src/` at HEAD `7c1ea6c3` (40 `.rs` files;
  32,478 LoC by `wc -l`, including `hover_tests.rs` 1,527 LoC).
- LSP protocol surface: `tower-lsp-server 0.23` (`tools/shape-lsp/Cargo.toml`
  line 21); LSP types from `tower_lsp_server::ls_types`.
- Parity target: rust-analyzer feature documentation
  (https://rust-analyzer.github.io/manual.html). Where a rust-analyzer feature
  is Rust-semantics-specific (e.g. lifetime inlay hints, `#[derive]` macro
  expansion, view-hir), it is flagged explicitly as **N/A for Shape** rather
  than counted as a parity gap.
- Out-of-scope: VSCode extension UX, tree-sitter highlighting, foreign LSP
  child-server protocol (covered tangentially via `foreign_lsp.rs` 2,551 LoC).

### Methodology

1. Inventory all LSP-protocol method handlers registered in
   `tools/shape-lsp/src/server.rs::ShapeLanguageServer::initialize`
   (lines 378-501) — these are the empirical truth of what shape-lsp claims to
   support at the protocol level.
2. For each `ServerCapabilities` field set, locate the handler implementation
   and read its top-level entry point to characterize **actual capability
   depth** (not just protocol registration).
3. For each rust-analyzer feature family (hover, completions, diagnostics,
   inlay hints, go-to-def, find-refs, rename, code-actions, semantic tokens,
   formatting, doc/workspace symbols, plus rust-analyzer-specific extras), map
   shape-lsp coverage to one of: COVERED, PARTIAL, MISSING, or N/A-for-Shape.
4. For every gap, estimate scope (SMALL / MEDIUM / LARGE) per the bulldozer
   ceiling-c convention; partition into proposed W2.2-N closure-waves with
   territory non-overlap.

### Audit binding (Reading 6 ground-truthing)

Every claim of presence/absence in §§1-2 is grep-verified at HEAD `7c1ea6c3`.
Concrete grep targets:

- Capability registration: `grep -n '_provider' tools/shape-lsp/src/server.rs`
  matches lines 386-498 inside `initialize`.
- Handler implementations: `tools/shape-lsp/src/server.rs` lines 617-1309 list
  every `async fn` LSP method.
- Per-family entry points cited verbatim in §1 table.

Where a feature is implemented but visibly shallow (e.g. text-based references
rather than AST-resolved cross-file), that is captured as PARTIAL with the
specific narrowing called out — NOT as COVERED.

## §5 Out-of-LSP-scope findings (bugs surfaced during audit; not r-a parity gaps)

These are shape-lsp behaviors that look broken or under-built but are NOT
rust-analyzer parity gaps. They are correctness/quality bugs that should be
filed separately.

1. **Find-references text-search false-positives on shadowing.**
   `definition.rs::find_all_references` (line 270) matches identifiers by
   word boundary regex within a single file. A shadowed inner-scope binding
   with the same name as an outer one will produce references to BOTH — the
   ScopeTree (`scope.rs`) infra exists to fix this but is not threaded
   through. Severity: MEDIUM. Categorized in W2.6 partition.

2. **Code-lens reference count uses same text-search.**
   `code_lens.rs::count_references` (line 293) has identical issue. Same fix
   in W2.6 cascades through.

3. **`workspace/symbol` silently scopes to open documents.**
   `server.rs:826-846` iterates `self.documents.all_uris()` (only documents
   the editor has opened). User cmd-T queries return INCOMPLETE results with
   no hint that this isn't the real workspace. Severity: MEDIUM. W2.7
   territory.

4. **`InlayHintConfig::default()` hardcoded — ignores client preferences.**
   `server.rs:1041` constructs `InlayHintConfig::default()` for every
   `inlay_hint` request, ignoring whatever the client passed via
   `workspace/configuration`. Most editors expose toggles like
   "Hide parameter hints"; shape-lsp respects none of them. Severity: SMALL.
   W2.4 territory.

5. **Quickfix triggers are diagnostic-message-text-keyed (`message.contains`).**
   `code_actions.rs:77, 101, 117, 133, 160, 192, 235, 265` all use
   `message.contains("...")` substring matches. Diagnostic message changes
   (i18n, rewording for clarity) silently break quickfix suggestions. The
   diagnostic `code` field IS set (`diagnostics.rs:62`); should key off
   `diagnostic.code` instead. Severity: MEDIUM (latent fragility, not user-
   facing today). W2.3 territory.

6. **`code_actions.rs::resolve_code_lens` is a no-op identity function.**
   `code_lens.rs::resolve_code_lens` (line 35) returns `lens` unchanged with
   the comment "Code lenses are already resolved in get_code_lenses". But
   `server.rs:491` registers `resolve_provider: Some(true)`. If the eager
   resolution is the design, the `resolve_provider` should be `false` to
   tell the client not to call `codeLens/resolve` at all. Either-or; current
   state wastes a roundtrip. Severity: SMALL polish.

7. **`completionItem.resolve_provider: false` ships all docs eagerly.**
   `server.rs:387`. Every completion item carries its full documentation. In
   a workspace with thousands of importable symbols, this bloats every
   `textDocument/completion` response. Severity: SMALL latent perf issue.
   W2.2 territory.

8. **`semantic_tokens.range: false` rejects range requests outright.**
   `server.rs:434`. For very large files, clients optimize by requesting
   only visible-viewport tokens. shape-lsp forces full-document parse and
   tokenization on every request. Severity: SMALL perf issue. W2.8
   territory.

9. **`call_hierarchy` operates on single file only.**
   `call_hierarchy.rs::incoming_calls` / `outgoing_calls` walk only
   `program.items` from the current file's text — cross-file callers /
   callees are not enumerated. Severity: MEDIUM. Same root cause as 1+2
   (no cross-workspace text index); fix lands as part of W2.7's workspace
   indexing.

These 9 items are tracked as **shape-lsp-quality-residuals** (analogous to
`shape-test-residuals-audit` per CLAUDE.md Known Constraints). All 9 are
either already absorbed into a W2.2-N closure-wave or are SMALL polish for
post-v0.3.
