1. **`@ai` annotation: marketing claim with no implementation and no book documentation** (§2.C). CLAUDE.md line 7 ("AI-first annotations") + line 125 (`@ai fn name(params) -> ReturnType {}` claim) + ADR-006 line 6716 + `docs/codebase-index/01-compilation.md:596-606` all describe a feature that has no grammar, no stdlib definition, no runtime handler, no book chapter. Disposition: surface to user — either populate `packages/ai/` + add `advanced/ai-annotations.mdx`, OR strip the "AI-native"/`@ai` references from CLAUDE.md and ADR-006.

2. **`tooling/polyglot.mdx:52` `DataTable | pd.DataFrame | Arrow IPC (zero-copy)` row is a STUB** (§2.B). `extensions/python/src/arrow_bridge.rs:17-34` shows both directions return `"not yet implemented"`. Page line 296 reinforces the false claim ("DataTable marshaling uses Arrow IPC for zero-copy bridging to pandas DataFrames"). Users following this row will get a not-yet-implemented error at runtime. **CRITICAL** for accuracy.

3. **`advanced/annotations.mdx:117-122` runtime `ctx` contract is wrong** (§1 row 2, §2.A). Book asserts `ctx.state` + `ctx.event_log` as the fixed-field set; actual `crates/shape-runtime/src/annotation_context.rs:88-100` has `cache` / `state` / `registries` / `events` / `data_range` + accessor-method API (`ctx.cache()` / `ctx.registry(name)` / `ctx.emit(event, data)` etc.). Book example at line 255 (`ctx["__impl"]`) further contradicts the "static typed object" framing. Load-bearing — every annotation author following the book gets the wrong mental model.

4. **`module` annotation target kind missing from book** (§2.A). Grammar `shape.pest:407-415` includes `module` in `annotation_target_kind`; book lists 6 of 7 kinds in both `comptime.mdx:127` and `annotations.mdx:80-89`. Functional gap — annotations targeting modules cannot be discovered by reading the book.

5. **Annotation import form divergence vs CLAUDE.md Known Constraint** (§2.D). `annotations.mdx:42` shows `from std::core::remote use { @remote }` named-import as the primary idiom; CLAUDE.md Known Constraint says named-import skips `Item::AnnotationDef` and only the namespace `use std::core::remote` form actually exposes annotations. 6. **TypeScript polyglot extension has no dedicated book chapter** (§2.B). `extensions/typescript/` is a first-party extension (CLAUDE.md crate map + `tooling/extensions.mdx:60`); book has `python-extension.mdx` (316 lines) but no `typescript-extension.mdx`. The polyglot mechanics page mentions `fn typescript` only obliquely (line 11 of extensions.mdx). Gap: TypeScript users have no equivalent of the Python extension chapter for install/Result<T>/marshalling/LSP/Neovim.

7. **`type_info` builtin removal not propagated to CLAUDE.md** (§2.A). `function_calls.rs:474-480` explicitly errors `"type_info has been removed. Use \`<expr>.type()\` for static type queries."`. The book correctly doesn't mention `type_info`. CLAUDE.md does (in the Language Features bullet for comptime builtins: "comptime builtins (`type_info`, `implements`, `warning`, `error`, `build_config`)"). Out-of-book-scope but in-audit-surface; CLAUDE.md cleanup needed.

8. **`set param name = expr` directive + `replace module (expr)` directive undocumented** (§2.A). Grammar `shape.pest:531, 537-538` defines both; `comptime.mdx:151-160` directive table covers neither. Minor but real gap.

9. **`LanguageRuntimeVTable.error_model` + `get_shape_source` vtable fields missing from polyglot capability table** (§2.B). `shape-abi-v1/src/lib.rs:704-714, 811, 813-828` defines both; `polyglot.mdx:244-249` table has 9 fields, omits these 2. `get_shape_source` is particularly load-bearing because it's the mechanism by which `python.*` / `typescript.*` namespaces become importable in Shape code.
