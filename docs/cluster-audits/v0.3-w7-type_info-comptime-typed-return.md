## §1 Current `type_info` API + return shape

### §1.1 Public `type_info(T)` builtin

**At HEAD: does not exist.** The bare name parses (it is a function-call expression with one argument) but the compiler hard-errors at `function_calls.rs:474`. There is no opcode, no `BuiltinFunction` enum variant, no runtime body, no schema registration. The call-site contract that downstream callers can rely on is THE COMPILE ERROR.

**Historical shape (pre-removal — recovered from vision docs).** The intended user-facing API per `docs/vision/distributed-comptime-async-vision.md:86` + `docs/vision/implementation-plan.md:357`:

```shape
comptime {
    let info = type_info(Currency)         // returns a type-reflection value
    // info exposes: name, kind, fields, variants, generic_params, ...
}
```

The v2-monomorphization-design (`docs/v2-monomorphization-design.md:264`) commits to "a typed struct, not a NaN-boxed object" — i.e. the return must be a fully typed compile-time value with statically known field offsets, not a dynamic dispatch carrier.

### §1.2 Replacement surface: `<expr>.type()` and `T.type()`

The replacement surface advertised by the removal error message is the universal-type-query method dispatch at `function_calls.rs:2045-2099`:

| Path | Receiver shape | Codegen | Runtime |
|---|---|---|---|
| Static (preferred) | Receiver type fully resolvable at compile time via `static_type_annotation_for_expr` (`type_ops.rs:564`) | Emits `OpCode::PushConst` of `Constant::TypeAnnotation(type_ann)`; pops the receiver if it has side effects | No runtime dispatch — the `TypeAnnotation` is baked into the bytecode constant pool |
| Type-symbol direct | `Point.type()` — receiver is a type identifier (`expr_is_type_symbol`, `type_ops.rs:556`) | Same as static; skips receiver value codegen (`function_calls.rs:2062-2065`) | Same — bytecode-baked |
| Runtime fallback | Type not statically resolvable (generic parameter, etc. — `should_runtime_type_query`) | Emits `OpCode::BuiltinCall(BuiltinFunction::TypeOf)` after pushing the receiver | **`todo!()` panic** at `vm_impl/builtins.rs:410-417` — Wave 5c migration pending |

**Consumed by:** Compile-time pattern dispatch over `Constant::TypeAnnotation` payloads. Examples:
- `crates/shape-vm/src/executor/builtins/type_ops.rs:570` (`Some(Constant::TypeAnnotation(ann)) => format!("{:?}", ann)`).
- `crates/shape-vm/src/executor/exceptions/mod.rs:309` (exception-handler type-match).
- `crates/shape-vm/src/compiler/patterns/checking.rs:89`, `binding.rs:61`, `helpers.rs:43/130` (pattern-match type guards).

**Shape of the consumed value:** `TypeAnnotation` (defined in `crates/shape-ast/src/ast/`) is the AST-level type representation — an enum with `Basic(String)`, `Reference(String)`, `Generic { name, args }`, `Function { params, returns }`, `Object(Vec<TypedField>)`, etc. It is NOT a runtime value type; it is an AST node embedded in the bytecode constant pool. Pattern dispatch over it happens at compile time, not at runtime.

### §1.4 Comptime-eval boundary

Comptime functions are registered via `register_typed_function` (`comptime_builtins.rs:156` for `implements`; line 200 for `warning`; line 249 for `build_config`). The signature is:

```rust
register_typed_function(
    &mut module,
    name,
    description,
    vec![],                            // arg types (TODO: comptime fns don't currently type-check args)
    ConcreteType::<RETURN_KIND>,        // declared return kind
    move |nb_args: &[KindedSlot], _ctx| -> Result<TypedReturn, String> { ... },
);
```

Return-shape options exposed via `ConcreteType` (the comptime-eval boundary's typed-return enum at `crates/shape-runtime/src/type_schema/` — referenced from `comptime_builtins.rs` at line 161/205/221/254):
- `Bool` (used by `implements`)
- `Unit` (used by `warning`, `error`, `__emit_*`)
- `Object` (used by `build_config` — returns a `TypedObjectStorage`-backed value; **this is the existing precedent for a struct-shaped comptime return**).

The `build_config` precedent is load-bearing: it shows that a comptime builtin returning a typed struct (with field offsets known at compile time, schema pre-registered via `register_predeclared_any_schema` at `comptime_builtins.rs:243-248`) is already a working pattern at HEAD, with the v2-raw TypedObjectStorage carrier dispatch fully wired (see the lengthy receiver-recovery comments at lines 274-313 — W17-comptime-vm-dispatch + Wave 2 Round 4 D4 ckpt-final-prime²).

## §2 Design tension

`type_info(T)` is a **comptime builtin** returning **type metadata**. Post-strict-typing (ADR-006 §2.7 NativeKind/HeapKind catalog; no `ValueWord` dynamic fallback), the canonical question is: **what carrier shape exposes type metadata to comptime user code with full static-type-safety + zero runtime tax + minimal architectural-surface debt?**

The five options below cover the design space. Each is evaluated in §3.

### Option (a): `Type` enum return

```shape
comptime {
    let t: Type = type_info(Currency)
    match t {
        Type::Primitive { name } => warning(f"primitive {name}"),
        Type::Struct { name, fields } => { for f in fields { ... } },
        Type::Enum { name, variants } => { for v in variants { ... } },
        Type::Generic { name, args } => { ... },
        // ... 10+ variants for full coverage
    }
}
```

Mirrors `SemanticType` (`semantic.rs:21`) directly as a user-facing Shape enum. Comptime user code pattern-matches over variants.

### Option (b): `TypeInfo` struct return

```shape
comptime {
    let info: TypeInfo = type_info(Currency)
    if info.kind == TypeKind::Struct {
        for field in info.fields {
            warning(f"field {field.name}: {field.type_info.name}")
        }
    }
}
```

Mirrors `shape_wire::metadata::TypeInfo` (`metadata.rs:35`) directly. Single struct with `Option<Vec<>>` fields for the variant-specific data. Comptime user code field-accesses.

### Option (c): Opaque handle + accessor functions

```shape
comptime {
    let t = type_info(Currency)              // opaque TypeHandle
    let name = type_name(t)                  // accessor fn
    let kind = type_kind(t)                  // accessor fn returns TypeKind enum
    let fields = type_fields(t)              // accessor fn returns Array<FieldHandle>
    for field in fields {
        warning(f"field {field_name(field)}: {type_name(field_type(field))}")
    }
}
```

Type metadata is an opaque value (single `HeapKind::TypeHandle` carrier); comptime user code goes through accessor functions that project to typed Shape values.

### Option (d): String-coded return

```shape
comptime {
    let info: string = type_info(Currency)   // e.g. "Struct{Currency, [code:string, rate:number]}"
    // parse/match the string
}
```

`type_info(T)` returns a serialized string. FFI-friendly (trivial to marshal); loses all static-type-safety; requires parser at every consumer.

### Option (e): Other — `comptime` macro / built-in pattern syntax

```shape
comptime {
    typematch Currency {
        struct { fields } => { for f in fields { ... } },
        enum { variants } => { for v in variants { ... } },
        primitive(name) => { ... },
    }
}
```

Introduce dedicated comptime syntax for type-reflection (not a function call returning a value). Sidesteps the carrier-shape question by making the type-shape inspection a language-level construct.

## §3 Per-option evaluation

### §3.1 Ergonomics

| Option | User-code shape | Discoverability | Verbosity for "iterate struct fields" |
|---|---|---|---|
| (a) Type enum | Pattern-match over 10+ variants; nested `match` for `Generic { base, args }` etc. | High — exhaustiveness check warns when new variants added | High — outer `match` + inner `for` |
| (b) TypeInfo struct | Field access (`info.kind`, `info.fields`); requires `if info.kind == X` guards before reading variant-specific `Option<>` fields | Medium — fields are explicit but `Option<>` discriminator-after-the-fact is awkward | Low — `if info.kind == Struct { for f in info.fields }` |
| (c) Opaque handle | Accessor function calls (`type_name(t)`, `type_fields(t)`); no direct field syntax | Low — must learn the accessor-fn surface; opaque type defeats LSP autocomplete on the value | High — every projection is an explicit fn call |
| (d) String-coded | String pattern-match or `parse_type_info(s)` to recover a structured shape | Very low — surface is opaque until parsed | Very high — must parse at every consumer |
| (e) Dedicated syntax | Compiler-level `typematch` block | Medium — new language surface to teach | Low — handles iteration directly |

**Winner (ergonomics): (b) — field access + `kind` discriminator is the lowest-friction shape for the common case ("iterate fields of a struct type"). Loses points for the `Option<>` field discipline; gains them back via the existing `build_config` precedent.**

### §3.2 Static-type-safety

| Option | Type-checked at compile time? | Exhaustiveness checking? | Refactor-safety |
|---|---|---|---|
| (a) Type enum | Yes — every variant fully typed | Yes — Shape's match exhaustiveness covers it | High — adding a variant errors all match arms |
| (b) TypeInfo struct | Yes — struct fields typed; `Option<Vec<FieldInfo>>` etc. | NO — `kind` discriminator + `Option<>` projections are not coupled at type level | Medium — adding a `TypeKind` variant doesn't error existing readers (silently returns `None` from the relevant `Option<>` field) |
| (c) Opaque handle | Partial — accessor return types are typed; opaque handle itself defeats compile-time projection | NO | Low — adding a type-shape adds an accessor fn; existing code doesn't error |
| (d) String-coded | NO — entire surface is `string` | NO | Very low — string format changes break consumers silently |
| (e) Dedicated syntax | Yes — compiler enforces exhaustiveness at the typematch block | Yes | Highest — language-level, full compiler reach |

**Winner (static-type-safety): (a) and (e). (a) wins via Shape's existing match-exhaustiveness machinery; (e) is theoretically equivalent but requires new grammar + new compiler dispatch (much larger surface).**

### §3.3 Comptime-eval cost

| Option | Construction cost (per `type_info(T)` call) | Reader cost (per accessor) | Refcount/Arc churn |
|---|---|---|---|
| (a) Type enum | Allocates an enum payload per variant (e.g. `Type::Struct { name: Arc<String>, fields: Arc<TypedArray<FieldInfo>> }`); construction reuses the registered SemanticType from inference | O(1) variant dispatch | Per-variant Arc allocation; equivalent to the existing inference-Type build |
| (b) TypeInfo struct | Single TypedObjectStorage allocation with pre-registered schema (mirrors `build_config` precedent at `comptime_builtins.rs:243-248`) | O(1) field access | One TypedObjectStorage refcount, plus nested TypedObjectStorage for FieldInfo (if `fields` populated) |
| (c) Opaque handle | One `HeapKind::TypeHandle` Arc allocation per `type_info()` call | O(1) accessor — but every accessor call rebuilds the projected value | Highest — every accessor allocates the projection |
| (d) String-coded | One `Arc<String>` allocation; serialization cost | O(string-length) on every parse | Low (single Arc); offset by repeated parse |
| (e) Dedicated syntax | Zero — pattern arms compile directly to compile-time-resolved code paths | Zero — no value materialized | Zero |

**Winner (comptime cost): (e) zero-cost > (b) single allocation + zero-cost field reads > (a) per-variant allocation > (c) per-accessor allocation > (d) parse-every-read.**

### §3.4 Interaction with existing callers

| Caller | (a) Type enum | (b) TypeInfo struct | (c) Opaque handle | (d) String-coded | (e) Syntax |
|---|---|---|---|---|---|
| `implements(T, Trait)` (`comptime_builtins.rs:156`) | Independent — implements takes type-NAMES (strings), no overlap | Independent | Could compose: `implements(type_name(t), Trait)` | Independent | Independent |
| `.type()` static path (`function_calls.rs:2058-2074`) | Could return a `Type` enum value — but currently returns `Constant::TypeAnnotation` (AST node) | Could return a `TypeInfo` struct — natural extension of the existing wire-TypeInfo projection | Could return an opaque handle — but loses the bytecode-baked benefit | Could return a string — already partially the case (`format!("{:?}", ann)` at `type_ops.rs:570`) | N/A — different surface |
| `.type()` runtime fallback (`BuiltinFunction::TypeOf`, currently `todo!()`) | Must construct a `Type` enum payload at runtime — needs full constructor wiring | Must construct a `TypeInfo` struct at runtime — single carrier; mirrors `build_config` | Must allocate handle | Format-string call | N/A |
| `comptime for field in type_info(T).fields` (vision-doc target — `distributed-comptime-async-vision.md:86`) | `match type_info(T) { Type::Struct { fields } => for f in fields { ... } }` — verbose | `for f in type_info(T).fields ?? []` — concise (assuming nullish-coalesce on Option) OR explicit guard | `for f in type_fields(type_info(T)) ?? []` | Requires parse | `typematch T { struct { fields } => for f in fields { ... } }` |
| `comptime for field in type_info(T).fields` UNROLL semantics | Compiler must teach the unroller about all 10+ variants | Compiler teaches the unroller about field access through Option<Vec<>> | Compiler teaches the unroller about the accessor-fn shape | Compiler must parse a string at unroll | Compiler owns the syntax directly |

**Interaction note:** The W7 audit is specifically about the RETURN SHAPE. The `comptime for` unroller (mentioned in `vision/implementation-plan.md:357` Sprint 8 line 7-8: "comptime for field in target.fields") is currently STUBBED at HEAD per `crates/shape-vm/src/compiler/expressions/misc.rs:710-743` (`compile_comptime_for` SURFACE outside comptime_mode, body deferred to phase-2c). Whatever return shape `type_info` lands with will dictate how the comptime-for unroller projects field iteration. This is a **downstream-implementation dependency** — the shape decision MUST come first.

## §4 Recommended shape

**Recommendation: Option (b) — `TypeInfo` struct return**, with careful adoption of a Shape-side mirror of `shape_wire::metadata::TypeInfo`.

### §4.1 Why (b) over the alternatives

- **Single-discriminator discipline (ADR-005 §1)**: A struct with a `kind: TypeKind` discriminator + Option<>-projected variant fields is structurally a single carrier (`HeapValue::TypedObject(Arc<TypedObjectStorage>)`), not 10+ parallel HeapKind variants. (a) Type enum would require either 10+ new HeapKind discriminators (every variant a separate type-confusion-safe carrier) OR a HeapKind::TypeEnum that wraps `Arc<TypeEnumPayload>` — the latter is fine but adds a new HeapKind with mid-cardinality just for type reflection. **(b) reuses the existing TypedObject machinery.**
- **`build_config` precedent at HEAD**: `comptime_builtins.rs:243-248` + 269-313 demonstrates the exact pattern — pre-register a typed schema, build via `typed_object_from_pairs`, project through `TypedObjectStorage` for the `ConcreteReturn::OpaqueTypedObject` shape. `type_info(T)` slots in as a more elaborate variant of the same pattern. **No new architectural surface.**
- **Direct projection of `SemanticType::to_type_info`**: The `SemanticType -> TypeInfo` projection (`semantic.rs:285-414`) is already implemented and tested (`semantic.rs:485-540`). `type_info(T)` is "call `SemanticType::to_type_info` at comptime + materialize as TypedObject". **The build-side is mostly already-written.**
- **`.type()` runtime-fallback path resolves cleanly**: `BuiltinFunction::TypeOf` (currently `todo!()` at `vm_impl/builtins.rs:410-417`) wires to "build the same TypeInfo TypedObject from the receiver's runtime-resolved type". One body, one shape. **No bytecode/runtime divergence.**
- **Wire-format symmetry**: `shape_wire::metadata::TypeInfo` is the snapshot/IO carrier. If `type_info(T)` returns the same shape (modulo Shape-vs-Rust type names), snapshot round-trip + remote-call type-metadata-passing comes for free.
- **Ergonomic-cost is bounded**: The "Option<> discrimination via kind guard" awkwardness (§3.1 medium-discoverability ding) is mitigated by Shape having pattern-matching over struct fields + nullish-coalescing + flow-sensitive narrowing (CLAUDE.md "Flow-sensitive narrowing" — `if info.kind == TypeKind::Struct { info.fields ... }` narrows `Option<Vec<FieldInfo>>` to `Vec<FieldInfo>` in the then-branch).

### §4.2 Why not (a) Type enum

- Costs 10+ new HeapKind discriminators OR one HeapKind::TypeEnum carrier with a payload enum. The former violates the "single discriminator per logical concept" preference; the latter is equivalent in carrier-shape to (b) but loses the wire-symmetry benefit.
- Adding a new type-shape (e.g. Trait objects, future Result/Option specializations) requires a new enum variant + cascade through every match arm at every consumer — vs (b) where adding a new `TypeKind` ordinal + optional struct field is local to the constructor.
- The match-exhaustiveness win is real but overstated: most `type_info` consumers only care about 2-3 shapes (Struct/Enum/Primitive); the rest go through a default arm anyway. Exhaustiveness becomes noise.

### §4.3 Why not (c) opaque handle

- Every accessor call allocates. Comptime is supposed to be zero-runtime-tax; opaque handle adds per-accessor Arc churn.
- LSP autocomplete on the handle value is broken — IDE can't see the type-shape through the opaque carrier.
- Comptime-for unroll over `type_fields(type_info(T))` is harder to teach the compiler than `type_info(T).fields` (the latter is field access on a known schema; the former is a generic-fn-call shape that must be specialized).

### §4.4 Why not (d) string-coded

- Static-type-safety: zero. Loses every benefit of strict typing.
- Parse-every-read cost.
- Refactor-hostile: changing the string format breaks every consumer silently.

### §4.5 Why not (e) dedicated `typematch` syntax

- New grammar; new compiler dispatch; new LSP teaching. Out-of-scope for a single sub-cluster fix.
- Sidesteps the carrier-shape question rather than answering it — the compiler still needs SOME internal representation of the type for the typematch arms to dispatch over. That internal representation is exactly the (a)/(b)/(c) question, just hidden from the user.
- COULD layer on top of (b) later as an ergonomic shortcut. Not mutually exclusive.

### §4.6 Recommended Shape-side `TypeInfo` shape

```shape
type TypeInfo {
    name: string,                              // "Currency", "Array<int>", etc.
    kind: TypeKind,                            // discriminator
    fields: Option<Array<FieldInfo>>,          // populated for struct/object types
    generic_params: Option<Array<TypeInfo>>,   // populated for generics
    variants: Option<Array<string>>,           // populated for enums (consider EnumVariant payload too)
    description: Option<string>,
}

enum TypeKind {
    Primitive,
    Array,
    Object,    // user-defined struct
    Table,
    Enum,
    Union,
    Function,
    Result,
    Optional,
}

type FieldInfo {
    name: string,
    type_info: TypeInfo,
    optional: bool,
    description: Option<string>,
}
```

Mirror of `shape_wire::metadata::TypeInfo` (`metadata.rs:35-78`) — same field names, same `Option<>` discipline, same `TypeKind` enum. Schemas pre-registered via `register_predeclared_any_schema` (mirror of the `build_config` precedent at `comptime_builtins.rs:243-248`).

### §5.2 Risks / cascade-site landmines

- **Recursive TypeInfo construction**: `TypeInfo.fields[i].type_info` is itself a `TypeInfo`. Recursive `TypedObjectStorage` construction has refcount-discipline implications (mirror of the W17-comptime-vm-dispatch + Wave 2 Round 4 D4 ckpt-final-prime² ceremony at `comptime_builtins.rs:274-313`). Every nested allocation needs proper `v2_retain` / drop pairing. **Land empirically with the same eprintln-bisect discipline as W6.**
- **Cyclic types** (e.g. `type Node { children: Array<Node> }`): naive recursion infinite-loops. Builder needs cycle-detection via type-id memoization.
- **Generic params at comptime**: `type_info(T)` inside a generic-fn body — does it return the concrete instantiation's TypeInfo or the parametric type variable? See Q-shape §6.
- **`SemanticType::to_type_info` already projects** to the Rust-side `TypeInfo` struct. The comptime builder essentially needs to convert that Rust struct into a Shape TypedObjectStorage with matching field names. Should be mostly mechanical.
- **Phase 2c stubs touching this territory**: `compile_comptime_for` (`misc.rs:710-743`) is currently SURFACE-and-stop outside `comptime_mode`. Its eventual implementation will consume `TypeInfo.fields`; the build of the TypeInfo carrier MUST happen first.

## §6 ADR-006 amendment shape

**ADR-006 amendment NOT strictly required.** The recommended (b) `TypeInfo` struct return uses **only the existing TypedObject machinery** — no new HeapKind variant, no new NativeKind variant, no new ABI shape. `TypeInfo` is just another user-facing type backed by `HeapValue::TypedObject(Arc<TypedObjectStorage>)` (HeapKind ordinal 1) with a pre-registered schema. This is the same shape as `build_config`'s return at HEAD (`comptime_builtins.rs:249-315`).

**ADR-006 amendment OPTIONAL** if we want to canonicalize the "comptime builtin returning a TypedObject with pre-registered schema" pattern as a named subsystem. Draft amendment text shape (if landed):

> §2.7.27 (or next-available) — Comptime-builtin TypedObject return carrier
>
> Comptime builtins (`build_config`, `type_info`, future comptime metadata builtins) that return user-visible structured data MUST use the `HeapValue::TypedObject(Arc<TypedObjectStorage>)` carrier with a pre-registered schema via `register_predeclared_any_schema`. The construction path is `typed_object_from_pairs` (or equivalent recursive variant for nested types); the receiver-recovery convention follows the §2.7.26 ModuleFn pattern (refcount discipline via `v2_retain` + Drop of the source kinded slot; outer `HeapValue::TypedObject(TypedObjectPtr)` wrapper takes a fresh share).
>
> This precludes returning comptime-builtin results via:
> - `ConcreteReturn::ValueWord` (deleted; refusal-on-sight)
> - Per-builtin new HeapKind variant (parallel-discriminator defection; ADR-005 §1)
> - `Box<HeapValue>` wrapper (§2.3 forbidden in new code)

**Recommendation:** Defer the amendment until the implementation lands. If the comptime-builtin-typed-return pattern accumulates 2+ more sites beyond `build_config` + `type_info`, then canonicalize. (Refuse #10 anti-deferral check passes: this is "defer the meta-amendment until the second site lands", not "defer the implementation".)

## §8 Q-shape — user decisions required before W7 implementation can dispatch

1. **Carrier shape: ratify (b) TypeInfo struct return, or pick (a)/(c)/(d)/(e)?** Recommendation in §4 is (b); user authorization required to bind downstream implementation.

2. **`type_info(T)` on generic type parameters at comptime — concrete or parametric?**
   - **Concrete-only:** `type_info(T)` inside `fn foo<T>()` errors unless T is a concrete instantiation at the call site (i.e. requires monomorphization to have already happened). Simpler; matches Rust's `std::any::type_name::<T>()` model.
   - **Parametric-supported:** Returns a TypeInfo with `kind: TypeKind::TypeVar` (or equivalent) + `name: "T"`. Lets comptime user code see generic params as opaque type-handles.
   - Recommendation pending: lean Concrete-only for v0.3 (matches the existing static-type-resolution shape in `static_type_annotation_for_expr`); Parametric is a follow-up if user code asks for it.

3. **`TypeInfo.fields` ordering — declaration-order or sorted?**
   - **Declaration-order:** Matches `TypedObjectStorage` field-order, matches user mental model.
   - **Sorted (alphabetical):** Deterministic for snapshot/wire stability.
   - Recommendation: declaration-order; the wire format already preserves this via `Vec<FieldInfo>` in `shape_wire::metadata::TypeInfo`.

4. **`TypeInfo` schema location — `std::core::types`, `std::reflection`, or top-level prelude?**
   - **`std::core::types`:** Co-located with existing type-system stdlib (cleanest namespace).
   - **`std::reflection`:** New namespace, signals intent.
   - **Top-level prelude:** Available everywhere without import.
   - Recommendation: `std::core::types` with prelude re-export; `TypeInfo` / `FieldInfo` / `TypeKind` are commonly-enough used in comptime code that prelude-availability is ergonomic, but having the canonical home in `core::types` is correct.

5. **(Optional) ADR-006 amendment landing — defer to second site, or land now?**
   - **Defer:** Per §6, the amendment becomes valuable once a 3rd "comptime builtin returning typed struct" site exists.
   - **Land now:** Codifies the pattern preemptively; teaches future agents the canonical shape.
   - Recommendation: defer; CLAUDE.md "Forbidden Patterns" already covers the "no ValueWord at runtime / no Box<HeapValue> wrapper" constraint that the §6 draft would otherwise restate.
