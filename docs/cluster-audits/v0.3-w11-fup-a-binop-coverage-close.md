# v0.3 W11-fup-A close — MIR BinOp coverage for Pow + bitwise

**Branch:** `worktree-agent-adf5e7da7b1a2bc1d` (parent `a8853958`).
**Closed:** 2026-05-18.
**Status:** CLOSE — MIR `BinOp` enum extended for the 6 operators the
W11-jit-new-array close §4 Class A identified; JIT consumer arms added in
the typed-codegen paths (`compile_binop_f64` / `compile_binop_int64` /
`compile_binop_i32_native`); honest surface-and-stop arms added in

## §0. Mission scope

W11-fup-A per dispatch: close the JIT MIR `Rvalue::BinaryOp` coverage gap
for the operators that fell through `lower_binary_op`'s `None` arm and
produced kind-blind `Rvalue::Aggregate(vec![l, r])` at
`crates/shape-vm/src/mir/lowering/expr.rs:1746-1756`. Per W11 close §4
Class A, the operators are: `Pow` (`**`), `BitAnd` (`&`), `BitOr` (`|`),
`BitXor` (`^`), `BitShl` (`<<`), `BitShr` (`>>`).

Empirical close gate (primary): W3.2-A snippet runnability **VM mode**
for the 8 Class A snippets should reach ≥8/8. Secondary: JIT-mode
improvement on the Pow / bitwise sites; non-BinOp residuals (BitNot
unary, InstanceOf, open-range, IndexAccess slice) surface-and-stop with
structured shape per their own follow-up sub-cluster ownership.

## §1. Root cause analysis

MIR `BinOp` enum at `crates/shape-vm/src/mir/types.rs:373-387` covered
only `Add/Sub/Mul/Div/Mod/Eq/Ne/Lt/Le/Gt/Ge/And/Or`. The remaining AST
`BinaryOp` variants fell through `lower_binary_op`'s `None` arm at
`crates/shape-vm/src/mir/lowering/helpers.rs:297-308`, producing
`Rvalue::Aggregate(vec![l, r])` at the `Expr::BinaryOp` call site —
which the JIT MIR consumer at `crates/shape-jit/src/mir_compiler/
rvalues.rs:145-176` surface-and-stops because the destination slot's
`ConcreteType` is `Void` (no MIR statement walker stamps a
binary-op-result type via the Aggregate shape).

The bytecode VM is unaffected: the bytecode compiler emits typed opcodes
directly from AST (`PowInt`/`PowNumber` at `opcode_defs.rs:317-322`;
`BitAndInt`/`BitOrInt`/`BitXorInt`/`BitShlInt`/`BitShrInt` at
`opcode_defs.rs:1860-1873`) without consulting the MIR `BinOp` enum at
that emission site. The MIR enum is consumed by:
- the JIT MIR-to-IR pipeline (`crates/shape-jit/src/mir_compiler/
  rvalues.rs::compile_rvalue`),
- producing-MIR kind inference (`crates/shape-jit/src/mir_compiler/
  types.rs::infer_rvalue_kind_with_projections`),
- the v2 native-arithmetic dispatch helpers
  (`compile_binop_f64`/`compile_binop_i32_native`/
  `compile_binop_int64`/`compile_binop_bool`/`compile_binop`).

Per ADR-006 §2.7.5 stamp-at-compile-time: the BinOp variant IS the proof
of operation kind; extending the enum is the principled fix per
W11-jit-new-array close §4 Class A disposition.

## §2. Files changed

### `crates/shape-vm/src/mir/types.rs:373-387`

`BinOp` enum extended with 6 new variants:
`Pow`, `BitAnd`, `BitOr`, `BitXor`, `BitShl`, `BitShr`. Doc-comment cites
W11-jit-new-array close §4 + the bytecode VM's existing typed-opcode
flavours; the comment also names the 5 operators that remain unhandled
(`FuzzyEqual`/`FuzzyGreater`/`FuzzyLess`/`NullCoalesce`/`ErrorContext`/
`Pipe`) per the same close doc's "different semantics" disposition.

### `crates/shape-vm/src/mir/lowering/helpers.rs::lower_binary_op`

6 new `ast::BinaryOp::{Pow,BitAnd,BitOr,BitXor,BitShl,BitShr} =>
Some(BinOp::*)` arms added; the previous `None` catch-all is split into
the 6 mapped arms + the remaining 6 unmapped arms (the 3 fuzzy + 3
non-arithmetic). Doc-comment cites the W11-jit-new-array close §4
disposition + the bytecode-emission counterpart.

### `crates/shape-jit/src/mir_compiler/rvalues.rs` (4 exhaustive matches)

1. **`compile_binop_f64`** (line 458) — exhaustive match. Added arms:
   - `BinOp::Pow` → `jit_pow_f64` FFI call (the helper pre-existed at
     `crates/shape-jit/src/ffi/v2_math.rs:302`; this sub-cluster
     registers it for JIT use).
   - `BinOp::BitAnd | BitOr | BitXor | BitShl | BitShr` →
     surface-and-stop with structured message (bitwise on `f64` has no
     semantic in Shape; the producer should have stamped Int64).

2. **`compile_binop_i32_native`** (line 514) — non-exhaustive (had `_ =>
   Err(...)` arm). Added explicit arms before the `_ =>`:
   - `BinOp::BitAnd` → Cranelift `band`
   - `BinOp::BitOr` → Cranelift `bor`
   - `BinOp::BitXor` → Cranelift `bxor`
   - `BinOp::BitShl` → Cranelift `ishl`
   - `BinOp::BitShr` → Cranelift `sshr` (arithmetic right-shift,
     matching VM's `BitShrInt` semantic per `opcode_defs.rs:1856-1857`).

3. **`compile_binop_int64`** (line 614) — non-exhaustive (had `_ =>
   compile_binop` fallthrough). Added explicit arms before the `_ =>`
   (which then became the `BinOp::And | BinOp::Or` arm to keep the
   pre-W11-fup-A behaviour):
   - `BinOp::BitAnd / BitOr / BitXor / BitShl / BitShr` → same
     Cranelift native instructions as the i32 path.
   - `BinOp::Pow` → `jit_pow_i64` FFI call (added in this sub-cluster
     at `ffi/v2_math.rs::jit_pow_i64`). VM/JIT divergence on i64 Pow
     overflow is documented (the VM kind-flips to f64 on overflow per
     `executor/arithmetic/mod.rs:151-160`; the JIT path does NOT
     replicate the kind-flip per §2.7.5 stamp-at-compile-time —
     separate follow-up `jit-pow-int-overflow-promotion` if observed).

4. **`compile_binop`** (line 670) — exhaustive match. Added arms:
   - `BinOp::Pow | BitAnd | BitOr | BitXor | BitShl | BitShr` →
     surface-and-stop with structured message (kind-untyped reach to
     these ops indicates a producing-MIR kind-tracker gap; the typed
     paths in `compile_binop_f64` / `compile_binop_int64` /
     `compile_binop_i32_native` are the intended targets).

### `crates/shape-jit/src/ffi/v2_math.rs` (new function `jit_pow_i64`)

### `crates/shape-jit/src/ffi_refs.rs`

`FFIFuncRefs` struct extended with two new fields: `pub(crate) pow_f64:

### Unary negation JIT NaN-box result (sub-residual, NOT new)

The post-fix JIT output for `op_arith.shape` includes `-9223372036854775798`
instead of the expected `-10` for the final `print(-a)` line. This is a
**pre-existing JIT unary-negation bug** on `NativeKind::Int64` slots —
the JIT's `compile_unop::Neg` arm produces a NaN-boxed bit-pattern
instead of the raw i64. Pre-fix this code path was never reached because
the function failed earlier on the Pow Aggregate surface; now that
W11-fup-A unblocks Pow, the downstream Neg bug is exposed. Not
introduced by W11-fup-A — surface-and-stop with structured shape:
**out-of-scope** for this sub-cluster; tracked as
`W11-followup-jit-unary-neg-int64`.
