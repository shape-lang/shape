## §1 Root-cause analysis of silent-no-output

### §1.1 Pre-fix execution path

```
shape run --mode jit <file>
  → bin/shape-cli/src/commands/script_cmd.rs::execute_file
    → run_engine (script_cmd.rs:1363-1412)
      → engine.execute(&mut JITExecutor, source) [shape-runtime::engine::execution::execute]
        → execute_with_options (execution.rs:232)
          → executor.execute_program(...)  [JITExecutor::execute_program at executor.rs:28-105]
            → compile bytecode (executor.rs:96-101)
            → execute_with_jit (executor.rs:108-509)
              → JITCompiler::compile_program_selective (compiler/program.rs:597)
                → preflight_instructions per-function + main-code (program.rs:610-687)
                  ↳ Err on AllocSharedModuleBinding / LoadSharedModuleBinding / StoreSharedModuleBinding
              → Cranelift codegen
              → foreign_bridge linking
              → jit_fn(&mut jit_ctx)  ← actual JIT execution
              → return_type_tag-based result extraction
                ↳ Err on RETURN_TAG_NANBOXED kind-source gap (W10 SURFACE)
```

Pre-fix: any `Err` from `execute_with_jit` propagated up to `script_cmd.rs:1331-1338`:

```rust
let response = match run_engine(engine, source, execution_mode, interrupt_flag).await {
    Ok(r) => r,
    Err(err) => {
        let runtime_error = engine.get_runtime_mut().take_last_runtime_error();
        print_shape_error(&err, runtime_error.as_ref());
        std::process::exit(1);
    }
};
```

`print_shape_error` writes to **stderr** + `std::process::exit(1)`. Under the
project's standard smoke-gate harness shape (`shape run --mode jit $file 2>/dev/null | tail -1`),
the stderr error message is dropped and stdout is empty because no JIT-emitted
`println!` ever fired — surface presents as "JIT silent-no-output with exit=1".

The original supervisor bisection report (2026-05-17) noted the smoke-gate
harness was further defected via `tail | echo EXIT=$?` capturing tail's exit
code instead of shape's, which masked the exit=1 signal and made the surface
look like "exit=0 + empty stdout" — the corrected harness shape uses `$?`
immediately after the pipeline.

### §1.2 Empirical anchor for the JIT compile-failure class

At workspace HEAD (`a8853958` pre-W12-fix), the only `vm_only_opcode_reason`
entries in `crates/shape-jit/src/compiler/accessors.rs:605-612` are:

```rust
match opcode {
    OpCode::AllocSharedModuleBinding
    | OpCode::LoadSharedModuleBinding
    | OpCode::StoreSharedModuleBinding => Some(
        "A.1C.3 outer-scope Shared module-binding opcode; Cranelift lowering pending",
    ),
    _ => None,
}
```

So the JIT preflight rejection class is narrow: only programs whose toplevel
main code reaches one of these three opcodes will fail `compile_program_selective`.
The four-opcode supervisor comment includes a fourth `DropSharedLocal` which
is gated alongside but elsewhere; the canonical empirical anchor used for
verification at this close is:

```shape
var counter = 100
fn read() { counter }
let val = read()
print(val)
```

Pre-fix `--mode jit` behavior: `Error: Runtime error: JIT compilation failed:
Main code contains unsupported constructs: JitPreflightReport { vm_only_opcodes:
[AllocSharedModuleBinding, LoadSharedModuleBinding], unsupported_builtins: [] }`
+ `exit(1)`. Under `2>/dev/null`: empty stdout.

VM mode runs cleanly and prints `100`.

### §1.3 The supervisor's `print(42)` claim was empirically over-broad

The supervisor prompt asserted:

> Even `echo "print(42)" > /tmp/p42.shape; ./target/debug/shape run --mode jit /tmp/p42.shape` → empty + (sometimes exit=0, sometimes 124 hang depending on commit).

This was NOT reproducible at HEAD `a8853958`. `print(42)` JIT-compiles cleanly:
the `print` builtin lowers to `jit_print_i64` (`crates/shape-jit/src/ffi/conversion.rs:289`),
which writes `42\n` to stdout via direct `println!`. The exit-code is 0; the
stdout is `42`.

The supervisor's hypothesis may have referred to an earlier HEAD or a debug
build with a 30s timeout cutoff — debug JIT compile of even a tiny program
takes ~40s in this NixOS environment, so `timeout 30` truncated the run before
the JIT-emitted `println!` fired. The architectural fall-through fix lands per
path (3) binding regardless of whether the specific `print(42)` reproducer
holds at HEAD.

Diff anchor (in `crates/shape-jit/src/executor.rs::JITExecutor::execute_program`,
right after `execute_with_jit`):

```rust
match self.execute_with_jit(engine, &bytecode, bytecode_compile_ms, emit_phase_metrics) {
    Ok(result) => Ok(result),
    Err(jit_err) => {
        let reason = jit_err.to_string();
        let function_name = "main".to_string();
        eprintln!(
            "[jit-fallback] function {function_name} failed JIT compile: \
             {reason}; running under interpreter"
        );
        tracing::info!(
            target: "shape_jit::fallback",
            function = %function_name,
            reason = %reason,
            "jit-fallback: function failed JIT compile, running under interpreter",
        );
        self.bytecode_executor.execute_program(engine, program)
    }
}
```

The `function_name` is hard-coded to `"main"` because the toplevel script
program does not have a function-level name; the diagnostic shape matches the
supervisor binding's `function X failed JIT compile` template. Per-function
fall-through inside `compile_program_selective` already lives in the existing
mixed-function-table architecture (functions that fail JIT compile are demoted
to `Interpreted` entries; the trampoline VM at `executor.rs:271-279` handles
the cross-mode dispatch) — the W12 fix specifically addresses the toplevel
program's all-or-nothing failure case.

The bytecode-compile-failure path at `executor.rs:96-101` (the
`compile_with_graph_and_prelude` Err arm) intentionally does NOT fall through.
Both `JITExecutor` and `BytecodeExecutor` use the same `compile_with_graph_and_prelude`
pipeline, so falling through on a bytecode-compile failure would just re-fail
in the interpreter. Bytecode-compile failures are user-facing source-level
errors and should surface to the CLI immediately.

## §6 ADR amendment decision

NONE. The W12 fix operates strictly at the `ProgramExecutor` trait boundary in
`crates/shape-jit/src/executor.rs::JITExecutor::execute_program`. The JIT
pipeline's internal kind-stamp invariants are unchanged:

- ADR-006 §2.7.5 (stamp-at-compile-time) preserved verbatim — no new
  runtime tag-bit decode, no new kind fabrication, no Bool-default extension
- ADR-006 §2.7.10 (Method-dispatch ABI Q11) preserved verbatim — no change
  to MethodFnV2 signature
- ADR-006 §2.7.11 (Value-call ABI Q12) preserved verbatim — no change to
  `op_call_value` / `call_value_immediate_*` shape
- §Forbidden Patterns preserved verbatim — no bridge/probe/helper/hop/translator/adapter/shim
  framing in any code or comment; no ValueWord / dynamic dispatch terminology

The fall-through dispatches to the existing `BytecodeExecutor::execute_program`
ProgramExecutor impl, which is the `--mode vm` execution path already documented.
The W12 fix specifically arranges for `--mode jit` to internally invoke that
same interpreter execution when the JIT pipeline fails, with a structured
diagnostic emission. This is bounded executor-path behavior, not an architectural
invariant amendment.
